## Stack decisions worth flagging to Stage 2+

1. **Next.js 15.5.15, not 16.** The prompt commanded
   `pnpm create next-app@latest`, which on this machine pulled Next
   16.2.4 — the locked stack says Next 15. I downgraded to the latest
   Next 15.x backport (`15.5.15`, dist-tag `backport`) to honour the
   lock. ESLint scaffold from Next 16 used the new
   `eslint-config-next/core-web-vitals` subpath imports, which Next 15
   does not export — replaced with `FlatCompat.extends("next/core-web-vitals", "next/typescript")`
   via `@eslint/eslintrc`. Stage 2+ may want to keep this shim until
   Next 16 is on the table.

2. **Tailwind v4 token wiring.** Brand colours live in plain `@theme`
   (so they emit both `:root --color-brand-*` *and* utilities). Fonts
   live in `@theme inline` with `--font-sans: var(--font-sans)` and
   `--font-bangla: var(--font-bangla)`, which is the canonical pattern
   when the next/font CSS variable shares a name with the Tailwind theme
   key — `inline` skips the :root emission step so there is no circular
   reference. Stage 2's `<BilingualText>` and `Prose` primitives can
   rely on the `font-sans` / `font-bangla` utilities resolving directly
   to the next/font-injected variables on `<html>`.

3. **Favicon + OG generation.** Both binary assets are produced by
   `scripts/build-assets.mjs`, which uses only Node built-ins
   (`zlib`, `Buffer`) — no headless browsers, no extra deps. Re-run it
   with `node scripts/build-assets.mjs` whenever the brand tokens or
   placeholder mark change. Output is committed so the asset pipeline
   does not depend on a postinstall step. The OG image is intentionally
   a placeholder; replace before launch (see launch checklist).

4. **`app/favicon.ico` removed.** create-next-app drops a stock favicon
   into `src/app/`, where Next auto-generates a `<link rel="icon">`.
   We deleted it so `/public/favicon.ico` (the spec-defined location)
   is the single source of truth. Browsers fall back to `/favicon.ico`
   by default; Stage 2 can re-add `app/icon.{svg,png}` if the team
   wants the auto-injected `<link>` tag.

5. **No `vercel.json`.** Vercel auto-detects `output: 'export'` for
   Next 15. Adding a `vercel.json` is not required and was skipped per
   the prompt's "only if needed" guidance.

6. **`pnpm-workspace.yaml` carries `ignoredBuiltDependencies` for
   `sharp` and `unrs-resolver`.** create-next-app added this; we kept
   it. Static export does not need `sharp` (we set
   `images: { unoptimized: true }`), and pnpm now skips the native
   build step on install. Harmless, but Stage 2+ should not remove it
   without re-checking install warnings.

7. **`noUncheckedIndexedAccess: true` is on.** This will bite Stage 2's
   loaders (`getEntrepreneur(id)` array indexing, `params.id` access,
   etc.) — every indexed access produces `T | undefined`. Plan for
   explicit guards or `??` defaults rather than `as` casts.

8. **`trailingSlash: true`** is set in `next.config.ts`. Stage 2's
   content loaders are unaffected, but Stage 4/5's `generateStaticParams`
   will produce URLs like `/podcasts/sample-ep01/` — the `index.html`
   pattern works on every static host, including Vercel and GitHub
   Pages, so no special config needed.

## Conditions encountered vs the prompt

- **`pnpm create next-app --no-turbopack`** is no longer a valid flag in
  the current CLI (Turbopack is opt-in via `--turbopack`; the absence of
  the flag *is* the no-turbo path). Behaviour is identical.
- **`V2V_Bridge_PRD_v1.0.docx`** is not present in this repository. Only
  `V2V_Bridge_PRD_Staged_v1.1.md` was available, which contains the
  technical scope Stage 1 needs. Stage 2 should pull the docx from
  whichever Drive folder holds the original if it needs the full
  functional PRD (Content Types & Data Model, Safeguarding sections).
- **Node version** on this machine is 24.11.0. The locked stack says
  Node 20 LTS. Stage 1 builds clean on Node 24; Vercel will use Node 20
  per its default runtime. Add an `"engines": { "node": "20.x" }` to
  `package.json` only if Stage 2+ wants to enforce it locally.
- **`git push`** was skipped. No `origin` remote is configured on this
  clone. Add the remote and push the branch when the GitHub repo is
  provisioned:
  ```
  git remote add origin <CapeC-org-repo-url>
  git push -u origin stage-1-foundation
  ```

  `text-brand-teal`, `bg-brand-coral`, `bg-brand-cream`, `text-brand-ink`,
  `text-brand-mute`, `border-brand-rule`. Plus the raw CSS vars

  The font variables are bound on `<html>` so any descendant works.
- A clean placeholder homepage (`src/app/page.tsx`) — Stage 3 replaces
  it. Layout (`src/app/layout.tsx`) is intentionally minimal; Stage 2
  fleshes it out with `<Header />` and `<Footer />`.
- The asset script `scripts/build-assets.mjs` is generic enough that
  Stage 6's per-page OG images can extend it (or copy the PNG writer)
  rather than pulling in `sharp` / `@resvg/resvg-js`.
